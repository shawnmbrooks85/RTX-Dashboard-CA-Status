01 - Overview & Architecture

What is this dashboard?

The RTX CA Certificate Status Dashboard is a read-only monitoring display for PKI certificate health in your Active Directory environment. It surfaces Enterprise CA certificate expiry dates and self-signed certificate inventory from the local machine's certificate stores, rendered as a real-time single-page web application.

Data Flow

Active Directory (certutil -TCAInfo)     Local Machine Cert Store
           ↓                                       ↓
  Collect-CACertData.ps1         (CryptoAPI / Get-ChildItem Cert:\)
  (runs every 60 min)
           ↓
  data/ca_data.json
           ↓
  serve.ps1  (HTTP web server on port 8089)
           ↓
  Browser  (auto-refreshes every 60 seconds)

The collector discovers Enterprise CAs via certutil -TCAInfo, pings each CA, retrieves expiry dates via certutil -CAInfo, and falls back to the local cert store for offline CAs. Self-signed certs are enumerated from Cert:\LocalMachine\My, CA, and Root.

02 - Technology Stack

Frontend (Zero Dependencies)

Plain HTML, CSS, and vanilla JavaScript - no React, Angular, or npm. No Chart.js required (certs display as tables, not charts).

  • index.html - Page structure, header, skeleton loader
  • app.js - Data fetching, polling, header update, theme cycling
  • cacert.js - Certificate table rendering, status pills, sort logic
  • styles.css - Shared RTX design system (identical to Admin/Kiosk dashboards)

Backend (PowerShell Only)

All server-side logic uses PowerShell 5.1. No IIS, no ASP.NET, no third-party frameworks.

  • scripts/Collect-CACertData.ps1 - CA discovery and cert data collector
  • scripts/serve.ps1 - Standalone HTTP/HTTPS web server (System.Net.HttpListener)
  • scripts/Setup-CACertDashboard.ps1 - 6-step deployment wizard
  • scripts/Remove-CACertDashboard.ps1 - Clean uninstall
  • scripts/Update-CACertDashboard.ps1 - File-only update (SHA256 diff)
03 - Certificate Discovery

Enterprise CA Discovery (Method 1 - certutil -TCAInfo)

Primary method. Runs certutil -TCAInfo which queries Active Directory for all published Enterprise CAs. Parses the output for Config: strings (format: SERVER\CA-Name).

  1. Output is split into blocks on blank lines
  2. Each block matching a backslash config string is extracted
  3. Config and display name are parsed via regex

Enterprise CA Discovery (Method 2 - COM Fallback)

If certutil fails or returns no results, falls back to the CertificateAuthority.Config COM object:

$caLocator = New-Object -ComObject CertificateAuthority.Config
$config = $caLocator.GetConfig(0)  # Returns first CA in AD

This requires the Certificate Services client RSAT feature or a machine enrolled in the domain.

CA Reachability Test

Each discovered CA is pinged via certutil -config <config> -ping. If the output contains interface is alive, the CA is marked reachable and full certificate details are retrieved.

Certificate Details - certutil -CAInfo

For reachable CAs, runs certutil -config <config> -CAInfo and parses:

FieldRegex Pattern
NotAfter (expiry)NotAfter\s*=\s*(.+)
Subject^\s*Subject\s*:\s*(.+)
Issuer^\s*Issuer\s*:\s*(.+)
ThumbprintCert\s*Hash.*?:\s*([0-9a-fA-F\s]+)

Certificate Details - Local Store Fallback

For unreachable CAs, the collector falls back to searching Cert:\LocalMachine\CA and Cert:\LocalMachine\Root for certificates whose Subject or Issuer matches the CA name. The most recently expiring match is used.

Self-Signed Certificate Enumeration

Scans three local machine stores - My, CA, Root - for certificates where Subject == Issuer. ConfigMgr/SMS-generated certs (SMS_SiteCode, SMSI) are excluded to reduce noise if the dashboard is co-hosted on a site server. Deduplication is done by thumbprint.

Thresholds are defined in the Get-CertStatus function in Collect-CACertData.ps1 and are easy to adjust.

Enterprise CA Summary Grid

Above the Enterprise CA table, a summary grid shows individual counts for each status category:

  • Total — total number of Enterprise CAs discovered
  • Valid — CAs with ≥ 60 days remaining
  • Warning — CAs expiring within 15–59 days
  • Critical — CAs expiring within 14 days
  • Expired — CAs with a past expiry date
  • Unreachable — CAs that did not respond to certutil -ping

Each count is color-coded (green / amber / red) and rendered in large bold numerals for at-a-glance visibility.

Reachability Indicator

Within the Enterprise CA table, each row shows a colored dot next to the CA name:

  • ? - ? (green) - CA responded to certutil -ping
  • ? - ? (red) - CA did not respond; cert data sourced from local store if available
05 - Data Collector

How Collect-CACertData.ps1 Works

  1. Discovers Enterprise CAs from Active Directory via certutil -TCAInfo
  2. Falls back to the CertificateAuthority.Config COM object if step 1 yields nothing
  3. Pings each CA via certutil -config <config> -ping
  4. For reachable CAs: retrieves expiry, subject, issuer, and thumbprint via certutil -CAInfo
  5. For unreachable CAs: searches local machine cert stores for a matching certificate
  6. Scans Cert:\LocalMachine\My, CA, Root for self-signed certs (Subject == Issuer)
  7. Calculates daysRemaining and assigns status for each cert
  8. Writes atomic JSON to data/ca_data.json via temp file + rename

JSON Output Structure

{
  "caCertExpiry": {
    "collectedAt": "2026-04-14T16:00:00Z",
    "enterpriseSummary": { "total":2, "valid":1, "warning":1, "critical":0,
                           "expired":0, "unreachable":0 },
    "selfSignedSummary": { "total":5, "valid":5, "warning":0,
                           "critical":0, "expired":0 },
    "cas": [
      { "caName":"EWAN-CA", "caConfig":"CM1\\EWAN-CA",
        "reachable":true, "status":"warning",
        "notAfter":"2026-05-20", "daysRemaining":36,
        "subject":"CN=EWAN-CA,...", "thumbprint":"ABC123..." }
    ],
    "selfSigned": [
      { "friendlyName":"IIS-SelfSigned", "store":"My", "status":"valid",
        "notAfter":"2027-01-01", "daysRemaining":262,
        "subject":"CN=IIS-SelfSigned,...", "thumbprint":"DEF456..." }
    ]
  }
}

Atomic JSON Write

The collector writes a temp file (ca_data_<random8>.tmp) and then moves it atomically to ca_data.json. This prevents the web server from serving a partially-written file. Up to 10 retries with 500ms delays are attempted if a file lock is held.

Scheduling

The data collector runs as a Windows Scheduled Task named CA-Dashboard-DataCollect:

  • Trigger: -RepetitionInterval 60 minutes (configurable at setup time)
  • Principal: SYSTEM by default (ServiceAccount logon); or a named domain account (Password logon) if specified during setup
  • Multiple instances: IgnoreNew (prevents overlap)
  • Action: powershell.exe -NonInteractive -ExecutionPolicy Bypass -File Collect-CACertData.ps1

60 minutes is the recommended default for this dashboard. CA certificate expiry is slow-moving data measured in months — hourly polling gives adequate freshness without unnecessary RPC overhead to the CA.

Requirements

  • Windows PowerShell 5.1
  • Domain-joined machine with network access to Enterprise CAs
  • RSAT Certificate Services or certutil available (built-in on all Windows Server installations)
  • SYSTEM account must be able to reach CA hosts on TCP 135 / dynamic RPC
06 - Web Server

How serve.ps1 Works

Uses a standalone System.Net.HttpListener - not IIS.

  1. Checks for an SSL certificate binding via netsh http show sslcert
  2. If bound, starts HTTPS; otherwise starts HTTP
  3. Listens on http://+:8089/ (or https://+:8089/)
  4. Serves files from the install directory with appropriate MIME types
  5. Exposes /api/whoami for the user identity badge

Scheduled Task

The web server runs as CA-Dashboard-WebServer:

  • Trigger: AtStartup
  • Execution time limit: 9,999 days (effectively unlimited)
  • Multiple instances: IgnoreNew
  • Runs hidden: -WindowStyle Hidden

Default Port

Default port is 8089. URL ACLs are registered for both HTTP and HTTPS:

netsh http add urlacl url=http://+:8089/ user=Everyone
netsh http add urlacl url=https://+:8089/ user=Everyone

A Windows Firewall inbound rule is created for TCP 8089 on Domain and Private profiles.

07 - HTTPS & SSL Certificates

Certificate Options

  1. Enterprise CA (recommended): Request-CACert.ps1 or New-DashboardCert.ps1 requests a trusted cert from your AD CS using the WebServer template.
  2. Self-Signed: New-DashboardCert.ps1 generates a self-signed certificate with SANs (hostname, FQDN, localhost, 127.0.0.1). Stored in Cert:\LocalMachine\My.

Setup will display a [SKIP] warning if New-DashboardCert.ps1 is not found in the source directory. Run with HTTP until SSL is configured manually.

SSL Binding

netsh http add sslcert ipport=0.0.0.0:8089 certhash=<THUMBPRINT> appid={...} certstorename=MY
netsh http add sslcert ipport=[::]:8089 certhash=<THUMBPRINT> appid={...} certstorename=MY

Both IPv4 and IPv6 bindings are created. Uses SHA256 thumbprints - FIPS-compliant.

Safety

  • Only binds to port 8089 - does not affect port 443, 8086, or 8087

Scheduled Task Account Prompt

Setup presents two interactive prompts before deployment begins — one for collection interval and one for the task account. Press Enter at each to accept the default.

  Collect interval in minutes [60]:          <-- press Enter for 60, or type a number
  Task account [SYSTEM]:                     <-- press Enter for SYSTEM, or type DOMAIN\Account
  Password for DOMAIN\Account: ********      <-- only shown if a named account was entered

The account is registered with LogonType = Password for both scheduled tasks (CA-Dashboard-WebServer and CA-Dashboard-DataCollect).

DC-hosted CA: If your Enterprise CA lives on a Domain Controller locked down to Domain Admin accounts, SYSTEM will be blocked from certutil RPC queries. Supply a DA-privileged service account at this prompt to resolve the issue.

Silent / Unattended Deployment

# Default SYSTEM account, 60-min interval (member server)
.\Setup-CACertDashboard.ps1 -Force

# Custom interval
.\Setup-CACertDashboard.ps1 -Force -CollectInterval 120

# Named account (DC-hosted CA or hardened environment)
.\Setup-CACertDashboard.ps1 -Force -TaskAccount "DOMAIN\SVC-CA-Reader" -TaskPassword "P@ssw0rd"

# Named account + custom interval
.\Setup-CACertDashboard.ps1 -Force -TaskAccount "DOMAIN\SVC-CA-Reader" -TaskPassword "P@ssw0rd" -CollectInterval 120

Using -Force bypasses all interactive prompts. SSL defaults to [1] Self-signed unless -Force is combined with a manual SSL pre-configuration. Passing -TaskAccount and -TaskPassword together enables fully unattended deployment with a named service account.

Prerequisites

  • Windows Server (domain-joined) with network access to Enterprise CA hosts
  • PowerShell 5.1+ (Windows PowerShell - NOT PowerShell 7 required)
  • Administrator privileges (scheduled tasks, firewall rules, URL ACLs)
  • certutil.exe available (built-in on all Windows; RSAT adds CA admin tools)
09 - Updating

Update-CACertDashboard.ps1

Pushes updated UI and script files without reconfiguring infrastructure:

  1. Compares SHA256 hashes between source and deployed files
  2. Copies only changed files
  3. Preserves serve.ps1 port and path configuration
  4. Restarts the web server scheduled task
  5. Triggers an immediate data collection run

Does NOT modify: scheduled task configuration, firewall rules, SSL certificates, or URL ACLs.

10 - Uninstall

Remove-CACertDashboard.ps1

Cleanly removes all dashboard components:

  • Scheduled tasks (CA-Dashboard-WebServer, CA-Dashboard-DataCollect)
  • Windows Firewall inbound rule for port 8089
  • SSL certificate bindings (port-specific via netsh http delete sslcert)
  • URL ACL reservations (netsh http delete urlacl)
  • Dashboard install directory and all files

Does NOT remove: SSL certificates from the cert store, or directory source files.

# Run as Administrator in PowerShell 5.1
& "C:\RTX-Dashboard-CA-Status\scripts\Collect-CACertData.ps1"

Output is displayed in the console - check for [WARN] lines indicating certutil failures or parse errors.

Verify Scheduled Tasks

Get-ScheduledTask -TaskName "CA-Dashboard-*" | Select-Object TaskName, State, @{n="LastRun";e={$_.LastRunTime}}

Check JSON Output

$json.caCertExpiry.enterpriseSummary
$json.caCertExpiry.cas | Select-Object caName, status, notAfter, daysRemaining

12 - Browser Personalization

Theme Persistence

The theme preference (Dark / Frosty / Light) is stored in localStorage under the key ca-dash-theme. It persists across reloads and browser restarts per-browser, per-profile. Clearing browser data resets to Dark.

ThemeBest For
Dark (default)NOC screens, dim rooms, reduced eye strain
